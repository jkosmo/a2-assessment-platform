# Internationalization (i18n) Baseline

This project supports three locales in MVP:

- `en-GB` (default fallback)
- `nb`
- `nn`

## Locale Resolution

For API requests, locale is resolved in this order:

1. `x-locale` header
2. `Accept-Language` header
3. `DEFAULT_LOCALE` environment variable
4. fallback to `en-GB`

The resolved locale is exposed in `GET /api/me` as `user.locale`.

## Frontend Translation Workflow

Participant UI strings are externalized in:

- `public/i18n/participant-translations.js`

To add or update text:

1. Add/update keys for `en-GB` first.
2. Add same keys in `nb` and `nn`.
3. Keep keys identical across locales.
4. Verify language switch in `/participant` using the language dropdown.

## Backend Translation Workflow

Backend user-facing messages are defined in:

- `src/i18n/messages.ts`

Locale helpers are in:

- `src/i18n/locale.ts`

If you add a new localized API message:

1. Add a new message key in `src/i18n/messages.ts`.
2. Provide value in all supported locales.
3. Use `t(locale, key, params)` in route/middleware.
4. Add or update tests for locale behavior.

## Content Authoring In Admin Workspace

For module content authored through `/admin-content`, text fields support two formats:

1. Plain text string
- Example:
```text
Submit a practical reflection with quality checks.
```

2. Inline locale JSON (all three locales)
- Example:
```json
{
  "en-GB": "Submit a practical reflection with quality checks.",
  "nb": "Lever en praktisk refleksjon med kvalitetskontroller.",
  "nn": "Lever ein praktisk refleksjon med kvalitetskontrollar."
}
```

Notes:
- Locale JSON is supported for module metadata, participant task/guidance text, prompt fields, and MCQ question fields.
- For MCQ, `stem`, each `options` item, `correctAnswer`, and `rationale` can each be plain text or locale JSON.
- If plain text is used, existing content translation lookup still applies where dictionary entries exist.
- If locale JSON is used, runtime locale selection resolves directly from that JSON value.

## Adding a New Locale

1. Add locale code to `SUPPORTED_LOCALES` in `src/i18n/locale.ts`.
2. Add locale option and labels in `public/i18n/participant-translations.js`.
3. Add translation entries for all keys in participant and backend message maps.
4. If required, allow locale in env schema (`DEFAULT_LOCALE`) in `src/config/env.ts`.
5. Add regression tests for:
   - explicit locale selection
   - fallback behavior
   - at least one localized API message
